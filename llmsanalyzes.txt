Tiered Recovery Score and Readiness CalculationsThe application evaluates recovery daily through a tiered, device-agnostic calculation model. The system automatically adjusts its algorithm based on the user's available hardware. If an athlete has an advanced wearable capable of tracking Heart Rate Variability (HRV), the application executes in its Full Tier, assigning a heavy 40% weight to HRV, 30% to sleep metrics, 10% to Resting Heart Rate (RHR), and 20% to subjective wellness. If the user only has a basic smartwatch, the Medium Tier redistributes the weights, assigning 30% to RHR, 30% to sleep, and 40% to subjective wellness. For users with no wearables, the Light Tier relies entirely on subjective ratings.Subjective wellness is calculated by normalizing daily ratings for muscle soreness, energy levels, mood, sleep quality, and perceived stress. For biometric calculations in the Full and Medium tiers, the system establishes a personalized baseline. Daily readings are evaluated using a standardized z-score method, comparing the daily metric against a 14-day moving average and standard deviation :$$z = \frac{x_{\text{Daily}} - \mu_{14}}{\sigma_{14}}$$The resulting z-score is mapped to a 0-100 scale. This ensures that the overall Recovery Score is fully personalized, accounting for individual physiological variations.Once calculated, the final score dictates the daily Readiness Status: "green" (optimal recovery, $\ge 70$), "yellow" (moderate fatigue, $40-69$), or "red" (overtraining risk, $< 40$). This status actively modifies the day's training session, scaling intensity and volume to match the athlete's current physical state.The APRE Computational ModelThe application's strength training engine utilizes the Autoregulatory Progressive Resistance Exercise (APRE) protocol, popularized by Dr. Bryan Mann. APRE dynamically adjusts training intensity on a set-by-set and week-by-week basis, matching an athlete's physical capability with the appropriate training stimulus.The system implements three distinct APRE protocols, each targeting a specific physical quality: the 10RM protocol for hypertrophy, the 6RM protocol for strength and hypertrophy, and the 3RM protocol for maximal strength and power. A standard APRE session consists of four sets, where the first two are warm-ups, the third is an "As Many Reps As Possible" (AMRAP) set, and the fourth adjusts load based on the third set's performance.Set 1 (Warmup): 10 reps @ 50% of estimated 6RM (Readonly)
Set 2 (Warmup):  6 reps @ 75% of estimated 6RM (Readonly)
Set 3 (AMRAP):   Max reps @ 100% of estimated 6RM (User enters reps)
Set 4 (AMRAP):   Adjusted reps @ Adjusted load (Computed using Set 3 reps)
The load for Set 4 is adjusted based on the number of repetitions completed in Set 3, utilizing the deterministic Mann table adjustments. The adjustment $\Delta w$ is added directly to the Set 3 load, and this adjusted Set 4 performance subsequently dictates the athlete's training maximum ($RM$) for the following week :$$\Delta w_{\text{NextWeek}} = f(\text{Reps}_{\text{Set4}})$$$$RM_{\text{NextWeek}} = RM_{\text{Current}} + \Delta w_{\text{NextWeek}}$$These mathematical adjustments are rounded to the nearest standard load increments, ensuring that all recommended weights map to real-world plate configurations.For bodyweight training, such as the customized calisthenics module, the system adapts the APRE model by substituting load adjustments with progression levels.   const dateIdx = headers.indexOf('Date');
  const sleepIdx = headers.indexOf('Sleep duration (h)');
  const rhrIdx = headers.indexOf('Resting heart rate (bpm)');

    records.push({
      date: fields[dateIdx].trim(),
      sleepHours: parseFloat(fields[sleepIdx]) || 0,

**Recommendation:** A phased JSX rewrite of `TodayPage.jsx` with extracted components (`HeroRing`, `SparklineCard`, `ExerciseList`, `WeeklyStrip`) would dramatically improve maintainability.

### 3. **Test coverage is uneven.**

It uses CommonJS `require()` and fails in the ESM project. The file still exists in the directory listing but `sessionLoad.test.ts` now covers its functionality. **Remove it** to avoid confusion.

The `ExerciseResult` type in `types.ts` has `repsPerSet: number[]` and `completedSets` — but the store's `handleToggleTraining` does NOT write these. ### 6. **`storage.ts` demo mode is fragile.**
The `_db()` function returns the active DB, but if `activateDemoData` is called after `init()`, the demo DB is created but `db` still refers to the real DB. The `getActiveDatabase()` export exists but isn't consistently used by `getAllSessions()`/`getAllCheckins()` — they use the closure-captured `db`.

**Recommendation:** Audit ALL data-access functions in `storage.ts` to use `getActiveDatabase()` instead of `db`.

### 7. **CSS is 70KB monolithic.**
All styles (base, pages, components, tour, onboarding, heatmap, APRE cards) live in one file. This makes it hard to find which component's styles are where, and increases the risk of specificity conflicts.

**Recommendation:** At minimum, extract component-specific CSS to co-located `.module.css` files. Vite supports CSS modules natively.

```typescript
// js/core/analytics.ts
export function calculateWeeklyCompletionRate(
  sessions: Session[],
  weekStart: Date
): number {
  const weekSessions = sessions.filter(s => isInWeek(s.date, weekStart));
  
  const totalPlanned = weekSessions.reduce((sum, s) => 
    sum + (s.exerciseResults?.reduce((es, er) => es + er.plannedSets, 0) ?? 0), 0
  );
  
  const totalCompleted = weekSessions.reduce((sum, s) => 
    sum + (s.exerciseResults?.reduce((es, er) => es + er.completedSets, 0) ?? 0), 0
  );
  
  if (totalPlanned === 0) return 1; // нет данных — не наказывать
  return Math.min(1, totalCompleted / totalPlanned);
}
```

### 3.1 CSV парсер — новый файл

```typescript
// js/core/import/csvParser.ts

export interface ParsedBiometrics {
  date: string;        // YYYY-MM-DD
  sleepHours: number;
  restingHR: number;
  hrv: number;
}

// Health Sync экспортирует CSV с заголовками
// Нужно определить реальные заголовки из твоего экспорта
// Типичные варианты: "Date", "Sleep duration (h)", "Resting heart rate (bpm)", "HRV (ms)"

export function parseHealthSyncCSV(csvContent: string): ParsedBiometrics[] {
  const lines = csvContent.trim().split('\n');
  if (lines.length < 2) return [];
  
  const headers = lines[0].split(',').map(h => h.trim().toLowerCase());
  
  // Flexible header matching — Health Sync меняет заголовки между версиями
  const findCol = (...candidates: string[]) => {
    for (const c of candidates) {
      const idx = headers.findIndex(h => h.includes(c.toLowerCase()));
      if (idx !== -1) return idx;
    }
    return -1;
  };
  
  const dateIdx = findCol('date', 'дата');
  const sleepIdx = findCol('sleep duration', 'сон', 'sleep hours');
  const rhrIdx = findCol('resting heart rate', 'rhr', 'пульс покоя');
  const hrvIdx = findCol('hrv', 'вариабельность');
  
  if (dateIdx === -1) throw new Error('Не найдена колонка даты в CSV');
  
  const records: ParsedBiometrics[] = [];
  
  for (let i = 1; i < lines.length; i++) {
    const fields = lines[i].split(',');
    if (fields.length < 2) continue;
    
    const record: ParsedBiometrics = {
      date: normalizeDate(fields[dateIdx]?.trim() ?? ''),
      sleepHours: sleepIdx !== -1 ? parseFloat(fields[sleepIdx]) || 0 : 0,
      restingHR: rhrIdx !== -1 ? parseInt(fields[rhrIdx], 10) || 0 : 0,
      hrv: hrvIdx !== -1 ? parseInt(fields[hrvIdx], 10) || 0 : 0,
    };
    
    if (record.date) records.push(record);
  }
  
  return records;
}

export async function mergeImportedBiometrics(
  records: ParsedBiometrics[],
  getAllCheckins: () => Promise<Checkin[]>
): Promise<{ updated: number; skipped: number }> {
  const existing = await getAllCheckins();
  const existingByDate = new Map(existing.map(c => [c.date, c]));
  
  let updated = 0;
  let skipped = 0;
  
  for (const record of records) {
    const existing = existingByDate.get(record.date);
    
    if (existing) {
      // Не перезаписывать если уже есть данные этого типа
      const needsUpdate = 
        (record.sleepHours > 0 && !existing.sleepHours) ||
        (record.restingHR > 0 && !existing.restHR) ||
        (record.hrv > 0 && !existing.hrv);
      
      if (needsUpdate) {
        await saveCheckin({
          ...existing,
          sleepHours: existing.sleepHours || record.sleepHours,
          restHR: existing.restHR || record.restingHR,
          hrv: existing.hrv || record.hrv,
        });
        updated++;
      } else {
        skipped++;
      }
    }
    // Если нет чекина за эту дату — создаём минимальный
    // (только биометрика, без субъективных данных)
  }
  
  return { updated, skipped };
}
```

```typescript
    include: ['wall slides', 'thoracic rotation', 'chest opener'],
    exclude: ['overhead stretch', 'behind-back stretch', 'shoulder extension']
  },
  back: {
    include: ['cat-cow', 'bird-dog', 'knee-to-chest'],
    exclude: ['loaded forward fold', 'extreme spine rotation', 'good morning stretch']
  },
  hips: {
    include: ['90/90 hip mobility', 'clamshells', 'pigeon pose modified'],
    exclude: ['deep loaded squat', 'extreme hip flexion', 'hurdler stretch']
  }
};

// В basePhase/buildPhase/peakPhase:
function filterStretchesForRehab(
  exercises: Exercise[], 
  rehabIssues: string[]
): Exercise[] {
  if (rehabIssues.length === 0) return exercises;
  
  const excludePatterns = rehabIssues
    .flatMap(issue => rehabStretchMap[issue]?.exclude ?? [])
    .map(p => p.toLowerCase());
  
  return exercises.filter(ex => 
    !excludePatterns.some(pattern => 
      ex.n.toLowerCase().includes(pattern)
    )
  );
}
```
